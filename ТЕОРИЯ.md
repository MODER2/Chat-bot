# Теория: как устроен и как работает чат-бот

Этот файл объясняет **как** написана программа, **почему** именно так и **что
означает** каждое обязательное понятие из задания: `Model`, `IBot`, `regex`,
`C4`, `UML`, `UI`. Читать можно сверху вниз — материал идёт от общей картины к
деталям.

---

## 1. Главная идея архитектуры

Программа разделена на **три независимых слоя**. Это классический приём: каждый
слой отвечает за своё и не лезет в чужое.

```
┌─────────────────────────────────────────────┐
│  UI  (интерфейс): окна JavaFX                │  ← что видит пользователь
├─────────────────────────────────────────────┤
│  ЛОГИКА БОТА: IBot, обработчики, regex        │  ← «ум» программы
├─────────────────────────────────────────────┤
│  MODEL + PERSISTENCE: данные и их хранение    │  ← сообщения и файл
└─────────────────────────────────────────────┘
```

**Зачем так?** Главное требование задания: *классы, обрабатывающие сообщения,
должны быть независимы от интерфейса*. Если логика не знает про окна, её можно
переиспользовать — например, подключить того же бота к Telegram. Проверить это
легко: в пакете `bot` нет ни одного `import javafx...`.

Структура пакетов (папок с кодом):

| Пакет | Слой | Что внутри |
|-------|------|-----------|
| `com.example.chatbot.model` | Model | `Message`, `Author`, `ChatHistory` |
| `com.example.chatbot.bot` | Логика | `IBot`, `AbstractBot`, `SimpleBot`, `BotContext` |
| `com.example.chatbot.bot.handler` | Логика | `MessageHandler` и обработчики команд |
| `com.example.chatbot.persistence` | Хранение | `HistoryStorage`, `FileHistoryStorage` |
| `com.example.chatbot.ui` | UI | `ChatApplication`, `LoginWindow`, `ChatWindow` |

## 2. Model — модель данных

**Model** — это слой данных: классы, которые описывают, *с чем* работает
программа, но не *как* её показывать. У нас это переписка.

- **`Author`** — перечисление (`enum`) из двух значений: `USER` и `BOT`.
  Перечисление выбрано потому, что авторов строго два — компилятор не даст
  ошибиться (в отличие от строки `"user"`, в которой легко сделать опечатку).

- **`Message`** — одно сообщение: автор + текст + время отправки. Класс
  **неизменяемый** (`final` поля, нет сеттеров) — такие объекты безопаснее.
  Здесь же два метода для файла:
  - `toStorageLine()` — превращает сообщение в строку `USER|2026-05-31 14:03:17|Привет`
    (это называется **сериализация**);
  - `fromStorageLine()` — обратное действие, восстановление объекта из строки
    (**десериализация**).

- **`ChatHistory`** — список всех сообщений **в оперативной памяти**. Умеет
  добавлять сообщения и считать статистику (сколько всего, сколько от
  пользователя, сколько от бота). Статистика нужна для команды «сколько сообщений?».

> «Хранение в оперативной памяти» = пока программа запущена, список живёт в RAM.
> Чтобы данные не пропали после выхода, их сохраняет слой Persistence (см. §6).

## 3. IBot — интерфейс бота

**`IBot`** — это **интерфейс**, то есть «контракт»: список методов, которые
обязан уметь любой бот, *без* указания, как именно они работают.

```java
public interface IBot {
    String respond(String userInput);          // ответить на сообщение
    void registerHandler(MessageHandler h);     // добавить новую команду
    List<MessageHandler> getHandlers();         // список команд
    String getBotName();                        // имя бота
}
```

**Зачем нужен интерфейс?** Чтобы окно чата зависело не от конкретного класса, а
от абстракции. Окну всё равно, кто перед ним — простой бот на правилах или
нейросеть, — лишь бы он реализовывал `IBot`. Это называется *программирование на
уровне интерфейсов* и делает код **слабосвязанным**: одну реализацию легко
заменить другой.

## 4. Абстрактные классы — AbstractBot и MessageHandler

**Абстрактный класс** — это незавершённый класс-заготовка: часть методов в нём
уже написана (общая для всех), а часть помечена `abstract` и обязательна к
реализации в наследниках. Задание прямо требует использовать абстрактные классы
«для описания программного интерфейса».

### 4.1. `AbstractBot`

Реализует `IBot` и пишет один раз общий алгоритм ответа:

```
respond(текст):
    для каждого обработчика по порядку:
        если обработчик.canHandle(текст):
            вернуть обработчик.respond(текст)
    вернуть «я не понял» (ответ по умолчанию)
```

Что именно подключить — решает наследник в абстрактном методе
`registerDefaultHandlers()`. Это паттерн **«Шаблонный метод» (Template Method)**:
скелет алгоритма в родителе, детали — в потомке.

### 4.2. `SimpleBot`

Конкретный бот. Всё, что он добавляет, — это список обработчиков:

```java
protected void registerDefaultHandlers() {
    registerHandler(new HelpHandler(this));
    registerHandler(new GreetingHandler());
    registerHandler(new CalculatorHandler());
    // ... и т.д.
}
```

### 4.3. `MessageHandler` — сердце расширяемости

Тоже **абстрактный класс**. Описывает один «обработчик одной темы». Общая часть
(работа с регулярным выражением) написана в нём, а наследники реализуют только
три метода: `getName()`, `getDescription()`, `respond()`.

Это реализация паттерна **«Стратегия» (Strategy)**: вместо гигантского
`if-else` у нас список взаимозаменяемых стратегий-обработчиков. **Чтобы научить
бота новой команде, не нужно менять старый код** — достаточно написать новый
класс-наследник и добавить одну строку `registerHandler(...)`. Это и есть
«гибкость», которую требует задание (принцип *Open/Closed* — «открыт для
расширения, закрыт для изменения»).

## 5. regex — регулярные выражения

**Регулярное выражение (regex)** — это шаблон для поиска текста по правилам.
Именно с его помощью бот «узнаёт» команды. В Java regex живёт в классах
`java.util.regex.Pattern` (скомпилированный шаблон) и `Matcher` (применение
шаблона к строке).

### 5.1. Базовые элементы, которые встречаются в проекте

| Запись | Что значит | Пример |
|--------|-----------|--------|
| `\d` | любая цифра 0–9 | `\d` → «7» |
| `\d+` | одна или более цифр | `12`, `157` |
| `\w` | буква/цифра/подчёркивание (с флагом Unicode — и кириллица) | `привет` |
| `\w*` | ноль или более таких символов | окончания слов |
| `\s` | пробельный символ | пробел, табуляция |
| `\s+` | один или более пробелов | разделитель слов |
| `\b` | граница слова | чтобы «привет» не сработал внутри слова |
| `( ... )` | **группа захвата** — запоминает кусок текста | числа в калькуляторе |
| `(?: ... )` | группировка **без** захвата | для альтернатив |
| `a|b` | «a ИЛИ b» | `на|и|от` |
| `?` | предыдущий элемент необязателен | `-?` — необязательный минус |
| `[.,]` | один символ из набора | точка или запятая |

### 5.2. Как работает распознавание

В базовом классе `MessageHandler`:

```java
this.pattern = Pattern.compile(regex,
        Pattern.CASE_INSENSITIVE | Pattern.UNICODE_CASE);
```

- `CASE_INSENSITIVE` — игнорировать регистр («Привет» = «привет»);
- `UNICODE_CASE` — чтобы это работало и для русских букв.

Метод `canHandle()` использует `matcher(input).find()` — ищет совпадение
**где-то** в строке (а не требует совпадения всей строки целиком).

### 5.3. Группы захвата на примере калькулятора

Команда «умножь 12 на 157» разбирается так (упрощённо):

```
(умнож\w*)  \s+  (-?\d+)  \s+(?:на|и)\s+  (-?\d+)
  глагол           число1                  число2
  группа 1         группа 2                группа 3
```

После `matcher.find()` мы достаём:
`m.group(1)` → «умножь», `m.group(2)` → «12», `m.group(3)` → «157».
Глагол превращаем в знак операции, числа — в `double`, считаем результат.
**Так реализуются «команды с параметрами».**

## 6. Persistence — сохранение истории в файл

Требование: *записывать историю в файл при завершении и загружать при запуске*.

- **`HistoryStorage`** — интерфейс с двумя методами: `save()` и `load()`. Опять
  абстракция: сегодня файл, завтра — база данных, и остальной код не изменится.
- **`FileHistoryStorage`** — реализация поверх текстового файла
  `chat_history.txt` в кодировке **UTF-8** (чтобы русский текст не «ломался»).
  Каждое сообщение — одна строка файла.

Когда что вызывается (см. `ChatApplication`):
- при **старте** → `storage.load()` наполняет `ChatHistory`;
- при **закрытии окна** и в методе `stop()` → `storage.save()` пишет всё в файл.

## 7. UI — графический интерфейс на JavaFX

**UI (User Interface)** — слой окон. Построен на **JavaFX** — это библиотека для
десктоп-интерфейсов. Базовые понятия JavaFX:

- **`Stage`** — окно операционной системы;
- **`Scene`** — «сцена», содержимое окна;
- **узлы (Node)** — элементы: `Label` (надпись), `Button` (кнопка),
  `TextArea` (поле ввода), `ListView` (список) и контейнеры компоновки
  (`VBox`, `HBox`, `BorderPane`).

Наши окна:

- **`ChatApplication`** — точка входа (`extends Application`). Метод `start()`
  задаёт порядок запуска: загрузить историю → показать окно имени → создать бота
  → открыть чат → сохранить при выходе.
- **`LoginWindow`** — стартовое модальное окно для ввода имени. «Модальное»
  значит, что до ввода имени к чату перейти нельзя (`showAndWait()` блокирует).
- **`ChatWindow`** — основное окно: список «пузырей» сообщений, поле ввода,
  кнопка «Отправить».

### 7.1. Требования к GUI и как они выполнены

| Требование | Где реализовано |
|-----------|-----------------|
| Окно ввода имени, потом основное окно | `LoginWindow` → `ChatWindow` |
| Шрифт и палитра, отличные от стандартных | `styles.css` (тёмная индиго-тема, шрифт Verdana) |
| Иконка приложения | `icon.png`, грузится в `ChatApplication.loadIcon()` |
| Отправка по горячей клавише | `ChatWindow.setupHotkeys()`: **Enter** и **Ctrl+Enter** отправляют, **Shift+Enter** — новая строка |

### 7.2. Почему логика отделена от UI

Окно `ChatWindow` **не знает**, как бот формирует ответ. Оно лишь вызывает
`bot.respond(text)` и показывает результат. Вся «умная» часть — в слое `bot`.
Это и есть выполнение требования о независимости обработки сообщений от интерфейса.

## 8. C4 и UML — диаграммы

Это два разных взгляда на архитектуру (обязательны по заданию):

- **UML (диаграмма классов)** — структура **кода**: какие классы есть и как они
  связаны (наследование, реализация интерфейса, агрегация). См. [docs/UML.md](docs/UML.md).
- **C4** — архитектура на **4 уровнях масштаба**: Context (система и её
  окружение) → Container (крупные части) → Component (модули) → Code (классы).
  См. [docs/C4.md](docs/C4.md).

Кратко: **UML отвечает на вопрос «из каких классов состоит код»**, а
**C4 — «как устроена система в целом, от пользователя до файла и интернета»**.

## 9. Полный путь одного сообщения (как всё связано)

Проследим, что происходит, когда пользователь пишет «умножь 12 на 157»:

1. **UI.** В `ChatWindow` пользователь жмёт Enter → вызывается `sendMessage()`.
2. **UI → Model.** Создаётся `Message(USER, "умножь 12 на 157")`, добавляется в
   `ChatHistory` и на экран.
3. **UI → Логика.** Вызывается `bot.respond("умножь 12 на 157")`.
4. **Логика.** `AbstractBot` перебирает обработчики; `CalculatorHandler.canHandle()`
   возвращает `true`.
5. **regex.** `CalculatorHandler.respond()` достаёт из текста числа и операцию
   через группы захвата, считает `12 * 157 = 1884`.
6. **Логика → UI.** Строка ответа возвращается в окно.
7. **UI → Model.** Создаётся `Message(BOT, "12 * 157 = 1884")`, добавляется в
   историю и на экран.
8. **Persistence.** При закрытии программы `FileHistoryStorage.save()` пишет всю
   историю (включая эти два сообщения) в `chat_history.txt`. При следующем
   запуске она снова загрузится.

## 10. Как добавить новую команду (практика расширения)

Допустим, нужна команда «подбрось монетку». Делается за 2 шага, **без изменения
существующего кода**:

**Шаг 1.** Новый класс в пакете `bot.handler`:

```java
public class CoinHandler extends MessageHandler {
    public CoinHandler() {
        super("(монетк\\w*|орёл\\s+или\\s+решк\\w*)"); // regex-триггер
    }
    public String getName() { return "Монетка"; }
    public String getDescription() { return "Подбросить монету: «подбрось монетку»"; }
    public String respond(String input, BotContext ctx) {
        return Math.random() < 0.5 ? "Орёл!" : "Решка!";
    }
}
```

**Шаг 2.** Одна строка в `SimpleBot.registerDefaultHandlers()`:

```java
registerHandler(new CoinHandler());
```

Готово. Команда автоматически появится и в ответе на «помощь», потому что
`HelpHandler` строит список динамически. Это наглядно показывает, ради чего была
выбрана такая архитектура.
